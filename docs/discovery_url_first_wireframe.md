# URL-first Discovery 화면 와이어프레임

이 문서는 Wedge의 첫 진입 UX를 **URL 입력 → Site Discovery → 추천 시나리오 → 짧은 온보딩 → 진단 시작** 흐름으로 그린 텍스트 와이어프레임이다.

## UX 원칙

1. 첫 화면에서는 URL만 묻는다.
2. 사용자가 먼저 시나리오를 고르게 하지 않는다.
3. Wedge가 Site Discovery / Preflight로 가능한 흐름을 찾고 추천한다.
4. 온보딩 질문은 추천 시나리오를 선택한 뒤에만 보여준다.
5. 시나리오별 질문은 1~2개로 제한한다.
6. 실제 결제, destructive action, OAuth/CAPTCHA 우회는 하지 않는다고 명확히 말한다.

---

## Step 1. URL 입력

첫 화면은 Google/ChatGPT처럼 **입력 하나에 집중**한다. 기능 설명, 안전 정책, 체크리스트는 첫 화면에서 길게 노출하지 않고 입력 이후 단계로 넘긴다.

```text
┌────────────────────────────────────────────────────────────────────┐
│                                                                    │
│                              Wedge                                 │
│                                                                    │
│           내 사이트에서 사용자가 막히는 순간을 찾아보세요           │
│                                                                    │
│          ┌──────────────────────────────────────────────┐          │
│          │ 분석할 사이트 URL을 붙여넣으세요...      →   │          │
│          └──────────────────────────────────────────────┘          │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 상태

- 사용자가 URL만 입력한다.
- 별도 CTA 버튼을 크게 두지 않고, 입력창 오른쪽의 `→` 또는 Enter로 시작한다.
- 첫 화면에서는 시나리오, device, goal, 안전 정책을 한 번에 묻지 않는다.
- 서비스 설명은 입력창 위 한 줄 카피로 제한한다.

### 대체 한 줄 문구 후보

```text
내 사이트의 전환 흐름을 점검해보세요
```

```text
어떤 사이트를 살펴볼까요?
```

```text
사용자가 어디서 망설이는지 찾아보세요
```

## Step 2. Discovery 진행 중

```text
┌────────────────────────────────────────────────────────────────────┐
│ Wedge                                                              │
│                                                                    │
│ 사이트를 살펴보고 있어요                                           │
│ https://example.com                                                │
│                                                                    │
│ ┌────────────────────────────────────────────────────────────────┐ │
│ │ 진행 중                                                        │ │
│ │                                                                │ │
│ │ ● 페이지 열기                                                  │ │
│ │ ● 첫 화면 확인                                                 │ │
│ │ ○ CTA 후보 탐색                                                │ │
│ │ ○ Form / Pricing / Contact 후보 확인                           │ │
│ │ ○ 추천 시나리오 정리                                           │ │
│ └────────────────────────────────────────────────────────────────┘ │
│                                                                    │
│ 이 단계는 전체 분석이 아닙니다.                                    │
│ 가능한 사용자 흐름을 찾기 위한 짧은 Preflight입니다.               │
└────────────────────────────────────────────────────────────────────┘
```

### 상태

- Discovery는 full analysis가 아니라 lightweight 탐색임을 명시한다.
- 사용자는 “뭔가 실행 중”임을 이해할 수 있다.
- 진행 단계는 실제 시스템 이벤트와 나중에 매핑 가능하다.

## Step 3. 추천 시나리오 결과

```text
┌────────────────────────────────────────────────────────────────────┐
│ Wedge                                                              │
│                                                                    │
│ 이 사이트에서 가능한 진단 흐름을 찾았어요                           │
│ https://example.com                                                │
│                                                                    │
│ ┌────────────────────────────────────────────────────────────────┐ │
│ │ 추천                                                           │ │
│ │ 첫 화면 CTA 점검                                               │ │
│ │                                                                │ │
│ │ 첫 화면에서 주요 CTA 후보 2개를 발견했어요.                    │ │
│ │ 사용자가 다음 행동을 바로 이해할 수 있는지 확인하기 좋습니다.  │ │
│ │                                                                │ │
│ │ 근거: hero section, primary button, nav CTA                    │ │
│ │                                                                │ │
│ │ [ 이 흐름으로 진단 ]                                           │ │
│ └────────────────────────────────────────────────────────────────┘ │
│                                                                    │
│ ┌────────────────────────────────────────────────────────────────┐ │
│ │ 가능                                                           │ │
│ │ 가입 / 문의 Form 점검                                          │ │
│ │                                                                │ │
│ │ 문의 또는 가입 form 후보 1개를 발견했어요.                     │ │
│ │ 입력 부담과 제출 전 신뢰 요소를 확인할 수 있습니다.            │ │
│ │                                                                │ │
│ │ 근거: contact link, email input candidate                      │ │
│ │                                                                │ │
│ │ [ 이 흐름으로 진단 ]                                           │ │
│ └────────────────────────────────────────────────────────────────┘ │
│                                                                    │
│ ┌────────────────────────────────────────────────────────────────┐ │
│ │ 낮음                                                           │ │
│ │ 구매 / 결제 흐름 점검                                          │ │
│ │                                                                │ │
│ │ 가격, 장바구니, 결제 진입점을 아직 찾지 못했어요.              │ │
│ │ 이 URL에는 랜딩 CTA나 문의 흐름이 더 적합해 보입니다.          │ │
│ │                                                                │ │
│ │ [ 그래도 직접 설정 ]                                           │ │
│ └────────────────────────────────────────────────────────────────┘ │
│                                                                    │
│ 원하는 흐름이 없나요?                                              │
│ [ 직접 시나리오 만들기 ]                                           │
└────────────────────────────────────────────────────────────────────┘
```

### 상태

- 추천도는 `추천 / 가능 / 낮음 / 사용 불가`처럼 사용자 언어로 표현한다.
- 내부 값은 `HIGH / MEDIUM / LOW / NOT_AVAILABLE`로 매핑한다.
- 낮은 추천도여도 사용자가 직접 진행할 수는 있다.

## 상태와 계약 매핑

| 화면 | 주요 계약 |
|---|---|
| URL 입력 | `POST /api/discoveries` |
| Discovery 진행 | `discovery_status_changed`, `discovery_checkpoint_created` |
| 추천 결과 | `GET /api/discoveries/{discoveryId}`, `ScenarioRecommendation` |
| 추천 선택 | `scenarioType`, `suggestedStartUrl`, `suggestedTarget` |
| 짧은 온보딩 | `ScenarioPlan.fit_requirements`, guided scenario option |
| 진단 시작 | `POST /api/runs` with `sourceDiscoveryId` |
| mismatch | `scenarioFitStatus=NOT_APPLICABLE`, `ScenarioMismatchReport` |

## 오늘 구현할 skeleton 범위

1차 skeleton에서는 아래까지만 구현한다.

- URL 입력 화면
- Discovery 진행 mock state
- 추천 시나리오 mock 카드
- 추천 선택 후 짧은 온보딩 mock
- 진단 시작 전 확인 화면 mock

API 연동은 다음 단계로 넘긴다.
